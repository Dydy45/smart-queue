**Objectif** : Prédire les temps d'attente en se basant sur l'historique réel des tickets, pas sur un `avgTime` statique configuré manuellement

## 📋 Vue d'ensemble

### Problème actuel

Aujourd'hui, l'estimation du temps d'attente est **statique et naïve** :

```
tempsAttente = position × service.avgTime
```

- `avgTime` est défini **manuellement** par l'OWNER lors de la création du service
- Aucune donnée réelle n'est collectée (pas de timestamps `calledAt`, `startedAt`, `finishedAt`)
- Aucun facteur contextuel n'est pris en compte (heure, jour, affluence, nombre de postes actifs)
- L'estimation ne s'améliore **jamais** avec le temps

### Solution proposée

Implémenter un système d'estimation **auto-apprenant** en 3 couches :

1. **Collecte de données** — Tracker les timestamps réels à chaque changement de statut d'un ticket
2. **Moteur d'estimation** — Algorithme statistique (moyenne pondérée glissante + facteurs contextuels) qui s'améliore avec l'historique
3. **Intégration UI** — Remplacer le calcul `position × avgTime` par les prédictions dynamiques + indicateur de confiance

### Approche technique : Statistical ML côté serveur (TypeScript)

**Pourquoi pas un vrai modèle ML (Python/TensorFlow) ?**
- La stack est 100% TypeScript/Next.js — pas de backend Python séparé
- Les données sont relativement simples (séries temporelles de durées)
- Une **moyenne pondérée glissante avec facteurs contextuels** (Exponentially Weighted Moving Average) donne des résultats quasi-équivalents à un modèle ML simple pour ce cas d'usage
- Zéro dépendance externe, zéro infrastructure supplémentaire
- Fallback gracieux vers `avgTime` quand il n'y a pas assez de données

### Bénéfices attendus

- ✅ **Précision** : Estimation basée sur des données réelles, pas une valeur fixe
- ✅ **Adaptation** : Le modèle s'ajuste automatiquement (heure de pointe, jour calme, nouveau staff)
- ✅ **Transparence** : Indicateur de confiance visible (peu de données → faible confiance)
- ✅ **Auto-correction** : L'OWNER n'a plus besoin de deviner le `avgTime`
- ✅ **Insights** : Dashboard de performance avec écart estimé vs réel

## 🏗️ Architecture Technique

### Données collectées par ticket

```
Ticket lifecycle:
  PENDING ──→ CALL ──→ IN_PROGRESS ──→ FINISHED
     │          │           │              │
  createdAt  calledAt   startedAt     finishedAt
     │          │           │              │
     ├──────────┤           ├──────────────┤
     waitDuration        serviceDuration
     (temps en file)     (temps de traitement)
```

### Features du modèle d'estimation

| Feature | Source | Description |
|---------|--------|-------------|
| `serviceDuration` historique | Tickets FINISHED | Durée réelle CALL→FINISHED par service |
| `hourOfDay` | `new Date().getHours()` | Heure courante (0-23) |
| `dayOfWeek` | `new Date().getDay()` | Jour de la semaine (0-6) |
| `activePostsCount` | Posts avec ticket IN_PROGRESS | Nombre de postes actifs simultanément |
| `queueLength` | Tickets PENDING count | Taille actuelle de la file |
| `recentThroughput` | Tickets FINISHED dernière heure | Débit récent (tickets/heure) |

### Algorithme : EWMA contextuel (Exponentially Weighted Moving Average)

```
estimatedServiceTime = α × recentAvg + (1-α) × historicalAvg

  où :
    α = 0.7  (poids fort sur les données récentes)
    recentAvg = moyenne des 20 derniers tickets FINISHED pour ce service
    historicalAvg = moyenne pondérée par créneau horaire (même heure, même jour)

estimatedWaitTime = (queuePosition / activePostsCount) × estimatedServiceTime

  Ajustements :
    - Si queuePosition > 10 → ajouter marge +10%
    - Si heure de pointe détectée → ajouter marge +15%
    - Si < 10 tickets historiques → fallback vers service.avgTime
```

### Fallback strategy

| Données historiques | Stratégie | Confiance |
|---------------------|-----------|-----------|
| 0 tickets FINISHED | `service.avgTime` (statique) | ⚪ Aucune donnée |
| 1-9 tickets | Moyenne simple des tickets existants | 🟡 Faible |
| 10-49 tickets | EWMA sans facteur contextuel | 🟠 Moyenne |
| 50+ tickets | EWMA complet avec contexte horaire/jour | 🟢 Élevée |
